# QR Code Instructions

How to create, place, and test the QR code used across the table package
(display board, table covering, both flyers, and the slideshow's final
slide).

> Course project draft — proposed program, not official Greece Central School
> District policy.

## Live URL

```
https://585swinemerchant.github.io/Work-Based-Learning-At-GCSD/
```

This is the site's home page and should be the destination for the primary
table QR code. All printed materials in this package should point to this
same URL so every scan lands in one consistent, well-organized place.

## Suggested QR Label

Pair the QR code with a short, action-oriented label rather than leaving it
unlabeled — visitors are more likely to scan when they know what they'll
get:

> **Scan to Explore** — WBL info, student pathway, forms & resources

Avoid vague labels like "Learn More" alone; naming what's on the other side
(forms, pathway info) sets clearer expectations.

## Generating the QR Code

Use any free, no-account QR generator that produces a static image (PNG or
SVG) — no login or paid service is required for this project. Enter the
exact live URL above, generate the image, and export at a high resolution
(at least 1000×1000px) so it can be scaled up for the display board or
table covering without pixelating.

## Placement Suggestions

- **Display board:** Panel 8 ("Scan to Explore") — large, centered, at a
  height easy to scan without crouching (roughly chest to eye level)
- **Table covering:** optional secondary placement on a front table-runner
  strip, as a backup if the display board is angled awkwardly or crowded
- **Flyers:** bottom of both the student/family and employer flyers, so a
  visitor who takes a flyer home still has the code
- Keep at least a small quiet margin (blank space) around every printed QR
  code — busy backgrounds or text placed too close can cause scan failures

## Accessibility Backup: Printed Short URL

Not every visitor has a working camera, a QR scanner app, or reliable data
access at the event. Print the plain-text URL beneath every QR code so it
can be typed in manually:

```
585swinemerchant.github.io/Work-Based-Learning-At-GCSD
```

This also matters for visitors who prefer not to scan unfamiliar codes —
seeing the actual destination domain in plain text builds trust.

## Testing Steps

Before the event:

1. Print a test copy of each material at final size (not just on-screen).
2. Scan the printed QR code with at least two different phones/scanner apps.
3. Confirm it opens the correct live site (not a typo'd URL or a stale
   cached version).
4. Check the code from a few feet away and at a slight angle, matching how
   a visitor would actually approach the table.
5. Re-test after any reprint — a resize or export step can occasionally
   corrupt a QR code's scannable pattern.

## Caution: QR Code Should Not Be the Only Access Point

Not every visitor will or can scan a QR code. The table should always work
for someone who never touches their phone: the display board and flyers are
written to stand on their own, with the QR code as a *bonus* path to more
depth — not a requirement for understanding what the program is. Keep a
staffed conversation as the primary access point, with the QR code and
printed URL as backups for taking information home.
