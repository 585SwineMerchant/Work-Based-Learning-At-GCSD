Our Commitment

Accessibility Statement

Working toward a site everyone can use — and telling you honestly where we are.

The GCSD Culinary Pathway intends for this website, and the Participant Resource Center that will be built alongside it, to be usable by the widest possible range of visitors, including people who use assistive technology such as screen readers, switch devices, or keyboard-only navigation.

Current Status

This site is in the planning stage, and a formal accessibility audit has not yet been completed. The pages produced under Work Package 2A were built to follow the practices below, but have not been tested against a third-party conformance standard or by users of assistive technology. Treat this statement as a description of intent and current practice, not a conformance claim.

What We Do Now

  • Every page includes a "Skip to main content" link for keyboard users
  • Page structure uses semantic headings (H1, H2, H3) in order, not styled text pretending to be a heading
  • Interactive elements (links, buttons, form fields) have a visible, high-contrast focus state
  • Color is never the only way information is conveyed — status labels use both color and text/shape
  • Images that convey meaning use alternative text; decorative marks are hidden from assistive technology
  • Layouts are responsive and usable at different zoom levels and screen sizes
  • Navigation is consistent from page to page

Known Limitations

  • No formal automated or manual accessibility audit has been completed against WCAG or another named standard
  • No testing has been done with actual assistive-technology users
  • Color contrast has not been independently verified against a formal ratio requirement
  • Pages produced in Work Package 2B (course, Gateway, and resource-center pages) have not yet been built or reviewed
  • Any document (Word or Excel) linked from a future Document Download Index may not yet be fully accessible in its native format

Getting Help

If you have trouble accessing information on this site, or need it in an alternate format, use the Accessibility Help route on the Contact, Help, and Concerns page. That route is currently a labeled placeholder pending a real, monitored contact.

Appendix

Accessibility Review Checklist

The checklist below is the internal review pass every new or revised public/participant page should go through before publication. It supports, but does not replace, a future formal accessibility audit.

Structure & Navigation

  • Page has exactly one H1, and headings descend in order without skipping levels
  • "Skip to main content" link is present and functions
  • Navigation order matches visual/reading order
  • Current page is indicated in navigation (aria-current)

Color & Contrast

  • Text and background colors meet at least a 4.5:1 contrast ratio for body text (3:1 for large text)
  • Status labels (Active/Proposed/Conditional/Exploratory) are distinguishable by shape/text, not color alone
  • Focus indicators are visible against every background used on the page

Keyboard & Interaction

  • Every interactive element is reachable and operable by keyboard alone
  • Tab order follows a logical reading sequence
  • No keyboard trap exists in any menu, form, or disclosure widget
  • Disclosure widgets (e.g., FAQ accordions) use native, accessible markup

Images, Media & Non-Text Content

  • Meaningful images have descriptive alternative text
  • Purely decorative images/icons are marked aria-hidden
  • Any future video/audio has captions or a transcript before publication

Forms & Status Communication

  • Every form field has a visible, programmatically associated label
  • Required fields and errors are announced to assistive technology, not shown by color alone
  • Status messages (e.g., placeholder/submission notices) use a live region where appropriate

Documents Linked From This Site

  • Linked Word/Excel documents open correctly and are labeled with file type and, where practical, size
  • Long PDFs (once produced) are checked for a tagged reading order before linking
  • Every document link's visible text describes its destination without relying on surrounding context alone

Review Sign-Off

Before a page moves from Internal Review to Approved for Planning Use per GOV-09, the Accessibility Lead confirms the checklist above and records the review date in the page's metadata block review-date field.
